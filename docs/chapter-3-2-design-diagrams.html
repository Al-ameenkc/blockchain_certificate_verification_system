3.2 Design of the Proposed System

This section presents structured design views for the Blockchain-Based Certificate Verification System: a normalized relational data model derived from an entity–relationship view, functional requirements as a use-case model, deployment-oriented architecture, high-level software structure, and representative activity workflows. Diagrams are rendered below using Mermaid so they can be inspected interactively in the browser and exported consistently for the project report.

3.2.1 Data Model (ERD → Relational Schema, 3NF)

The conceptual model separates reference-keyed certificates, audit activities, and small lookup entities so repeating domain values are not duplicated across many rows. The diagram shows cardinalities; the tables that follow translate the ERD into relational schemas. 

erDiagram
  DEPARTMENTS ||--o{ CERTIFICATES : issued_for
  DEGREE_CLASSES ||--o{ CERTIFICATES : classified_as
  CERTIFICATE_ACTION_KINDS ||--o{ CERTIFICATES : record_kind
  CERTIFICATE_ACTION_KINDS ||--o{ ACTIVITIES : logged_kind
  DEPARTMENTS {
    uuid dept_id PK
    string dept_name
  }
  DEGREE_CLASSES {
    uuid class_id PK
    string class_name
  }
  CERTIFICATE_ACTION_KINDS {
    string kind_code PK
    string description
  }
  CERTIFICATES {
    string reference_id PK
    string student_name
    string matric_number
    uuid dept_id FK
    uuid class_id FK
    date date_issued
    string kind_code FK
    datetime created_at
  }
  ACTIVITIES {
    uuid activity_id PK
    string kind_code FK
    string action_title
    text details
    string url
    datetime logged_at
  }

Relational schemas (logical / physical mapping)

Third Normal Form (3NF) is satisfied because: (1) every attribute depends on the whole primary key; (2) non-key attributes do not depend on other non-key attributes (department name is stored only in DEPARTMENTS, honours label only in DEGREE_CLASSES); (3) repeating categorical values for certificate disposition are centralised in CERTIFICATE_ACTION_KINDS. Optional migration note: the implemented prototype may store department and class strings directly on certificates for speed; normalisation here documents the target relational shape for production.

  • 1NF: Atomic columns; each certificate row describes one reference.
  • 2NF: No partial dependencies on composite keys (PKs are single-attribute).
  • 3NF: No transitive dependencies (names flow via FK joins, not duplicated facts).

Table specifications

Table Columns (PK/FK noted) Integrity / notes
departments dept_id PK (UUID), dept_name UNIQUE NOT NULL Lookup list aligned with institutional faculties/programmes.
degree_classes class_id PK (UUID), class_name UNIQUE NOT NULL Honours enumerations shared by many certificates.
certificate_action_kinds kind_code PK (TEXT), description NOT NULL Rows such as ISSUE, REGISTER; replaces duplicated free-text semantics at scale.
certificates reference_id PK; student_name, matric_number; dept_id FK→departments; class_id FK→degree_classes; date_issued; kind_code FK→certificate_action_kinds; created_at Mirrors anchored on-chain facts for dashboards; uniqueness on reference_id matches contract-level uniqueness intent.
activities activity_id PK; kind_code FK nullable; action_title; details; url; logged_at TIMESTAMPTZ NOT NULL Audit-style event stream; kind_code links structured logs to the same vocabulary as certificates where applicable.

On-chain storage is modelled separately: the smart contract maps reference_id→ structured certificate fields. It is not a SQL table, but it participates in the overall distributed data model as the tamper-evident system of record for verification reads.

3.2.2 Functional Requirements (Use Case Diagram)

Primary actors are the Administrator (authenticated issuer) and the Public Verifier. The Blockchain Network acts as a supporting subsystem for anchoring and tamper-evident reads; Supabase supports optional mirrored persistence and audit listings.
